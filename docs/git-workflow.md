# Git- und PR-Workflow

## Branch-Strategie
- Stabiler Hauptbranch: `main`
- Keine direkten Pushes auf `main`
- Arbeit immer in separaten Branches

## Branch-Namen
- `feature/repository-setup`
- `feature/backend-ticket-model`
- `feature/frontend-board`
- `fix/drag-drop`
- `docs/readme-update`

## Pull-Request-Regeln
- Merge nur per Pull Request
- Mindestens 1 Review durch den zweiten Entwickler
- Empfehlung: `Squash and merge`
- Vor Merge kurz lokal testen

## Mini-PR-Checkliste
- Projekt baut lokal
- Keine versehentlichen IDE-Dateien im Commit
- README/Doku angepasst, wenn nötig
- Ticket bzw. Aufgabe im PR kurz beschrieben
